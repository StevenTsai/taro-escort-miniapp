# 陪诊师收入查询功能需求文档

## 1. 需求概述

### 1.1 功能背景
当前系统已实现陪诊师展示、服务预约、订单管理等核心功能，但缺乏陪诊师收入查询功能。陪诊师无法直观了解自己的月度收入情况，包括总金额、已到账金额、未到账金额以及每笔收入明细。

### 1.2 功能目标
- 提供陪诊师月度收入概览
- 展示每笔陪诊订单的收入明细
- 区分已到账和未到账金额
- 支持按月份查询历史收入
- 提供良好的用户体验和数据可视化

### 1.3 适用角色
- 陪诊师（主要用户）

## 2. 功能需求

### 2.1 收入概览功能
- **功能描述**：展示陪诊师当月的收入统计信息
- **显示内容**：
  - 当月总收入金额
  - 已到账金额
  - 未到账金额
- **交互设计**：
  - 默认展示当前月份数据
  - 支持月份切换查看历史数据
  - 数据实时更新

### 2.2 收入明细功能
- **功能描述**：展示陪诊师当月每笔订单的收入详情
- **显示内容**：
  - 订单ID
  - 服务日期
  - 服务类型
  - 服务金额
  - 到账状态（已到账/未到账）
  - 到账时间
- **交互设计**：
  - 支持按到账状态筛选
  - 支持点击查看订单详情
  - 支持下拉刷新获取最新数据

### 2.3 月份切换功能
- **功能描述**：允许陪诊师查看不同月份的收入数据
- **交互设计**：
  - 提供月份选择器（下拉选择或滑动切换）
  - 支持查看最近12个月的数据
  - 切换月份时自动刷新收入概览和明细

## 3. 非功能需求

### 3.1 性能要求
- 页面加载时间：≤ 2秒
- 数据刷新时间：≤ 1秒
- 支持同时访问量：≥ 1000人

### 3.2 安全性要求
- 收入数据仅对对应的陪诊师可见
- 数据传输采用HTTPS加密
- 接口请求需携带有效的身份认证信息

### 3.3 可用性要求
- 页面可用性：≥ 99.9%
- 操作流畅，无明显卡顿
- 适配不同尺寸的手机屏幕

### 3.4 可靠性要求
- 数据准确性：100%
- 系统容错能力强，异常情况下给出友好提示

## 4. 数据需求

### 4.1 收入数据模型

#### 4.1.1 收入概览数据
```javascript
{
  "month": "2025-12",          // 月份，格式：YYYY-MM
  "totalAmount": 15000,         // 总收入，单位：元
  "receivedAmount": 12000,      // 已到账金额，单位：元
  "pendingAmount": 3000         // 未到账金额，单位：元
}
```

#### 4.1.2 收入明细数据
```javascript
{
  "id": "order_001",            // 订单ID
  "serviceDate": "2025-12-15",  // 服务日期，格式：YYYY-MM-DD
  "serviceType": "普通陪诊",     // 服务类型
  "amount": 500,                 // 服务金额，单位：元
  "status": "received",         // 到账状态：received（已到账）、pending（未到账）
  "receivedTime": "2025-12-16 10:30:00",  // 到账时间，格式：YYYY-MM-DD HH:mm:ss
  "patientName": "张三",         // 患者姓名
  "hospital": "北京协和医院"     // 服务医院
}
```

### 4.2 数据来源
- 订单系统：提供服务订单数据
- 支付系统：提供到账状态和时间
- 用户系统：提供陪诊师和患者信息

## 6. 接口设计

### 6.1 获取陪诊师月度收入概览
- **接口地址**：`/api/escorts/income/overview`
- **请求方式**：GET
- **请求参数**：
  | 参数名 | 类型 | 必选 | 说明 |
  |--------|------|------|------|
  | month | string | 否 | 月份，格式：YYYY-MM，默认当前月份 |
  | escortId | number | 是 | 陪诊师ID（从token中获取） |

- **响应数据**：
  ```javascript
  {
    "code": 0,
    "message": "success",
    "data": {
      "month": "2025-12",
      "totalAmount": 15000,
      "receivedAmount": 12000,
      "pendingAmount": 3000
    }
  }
  ```

### 6.2 获取陪诊师月度收入明细
- **接口地址**：`/api/escorts/income/details`
- **请求方式**：GET
- **请求参数**：
  | 参数名 | 类型 | 必选 | 说明 |
  |--------|------|------|------|
  | month | string | 否 | 月份，格式：YYYY-MM，默认当前月份 |
  | status | string | 否 | 到账状态：received（已到账）、pending（未到账），默认全部 |
  | page | number | 否 | 页码，默认1 |
  | pageSize | number | 否 | 每页条数，默认10 |
  | escortId | number | 是 | 陪诊师ID（从token中获取） |

- **响应数据**：
  ```javascript
  {
    "code": 0,
    "message": "success",
    "data": {
      "list": [
        {
          "id": "order_001",
          "serviceDate": "2025-12-15",
          "serviceType": "普通陪诊",
          "amount": 500,
          "status": "received",
          "receivedTime": "2025-12-16 10:30:00",
          "patientName": "张三",
          "hospital": "北京协和医院"
        }
      ],
      "total": 20,
      "page": 1,
      "pageSize": 10
    }
  }
  ```

## 7. 技术实现方案

### 7.1 前端实现
- **页面组件**：新增`escortIncome`页面组件
- **状态管理**：使用Zustand创建`useEscortIncomeStore`管理收入数据
- **数据请求**：使用现有`@/utils/request`工具封装API请求
- **UI组件**：使用Taro UI或自定义组件实现页面布局

### 7.2 后端实现
- **新增接口**：实现上述两个收入查询接口
- **数据处理**：从订单系统和支付系统获取数据，进行统计和处理
- **权限控制**：确保陪诊师只能访问自己的收入数据

### 7.3 数据库设计
- **无需新增表**：可基于现有订单表和支付记录表进行数据统计
- **索引优化**：为订单表的`escortId`和`serviceDate`字段添加索引，提高查询效率

## 9. 风险评估

### 9.1 风险识别
- **数据准确性风险**：订单数据和支付数据不同步导致收入统计错误
- **性能风险**：大量订单数据查询导致页面加载缓慢
- **权限风险**：陪诊师可能访问到其他陪诊师的收入数据

### 9.2 风险应对
- **数据准确性**：定期同步订单和支付数据，建立数据一致性校验机制
- **性能优化**：实现分页查询，添加数据缓存，优化数据库索引
- **权限控制**：严格的接口权限验证，确保陪诊师只能访问自己的数据

## 2. API接口设计

### 2.1 患者信息管理接口

#### 2.1.1 创建患者
- **接口路径**：`/api/patients`
- **请求方法**：`POST`
- **请求头**：
  - `Content-Type`: `application/json`
  - `Authorization`: `Bearer {token}`
  - `X-Request-ID`: 可选，请求唯一标识符，用于日志追踪
- **权限要求**：医生、护士、管理员角色
- **请求参数**：
```json
{
  "name": "张三",
  "gender": 1,
  "age": 65,
  "height": 175.00,
  "weight": 70.00,
  "phone": "13800138000",
  "histories": [
    {
      "type": 1,
      "content": "高血压病史10年",
      "record_date": "2024-01-01"
    },
    {
      "type": 2,
      "content": "青霉素过敏",
      "record_date": "2024-01-01"
    }
  ]
}
```
- **参数说明**：
  - `name`：患者姓名（必填，字符串，最大长度50）
  - `gender`：性别（必填，数字，1-男，2-女）
  - `age`：年龄（必填，数字，1-120）
  - `height`：身高（可选，数字，单位cm，保留两位小数，范围50-250）
  - `weight`：体重（可选，数字，单位kg，保留一位小数，范围10-200）
  - `phone`：联系电话（必填，字符串，11位手机号码，格式：1[3-9]\d{9}）
  - `histories`：基础病史列表（可选，数组，最多10个病史条目）
    - `type`：病史类型（必填，数字，1-既往病史，2-过敏史，3-家族病史）
    - `content`：病史内容（必填，字符串，最大长度5000，支持HTML格式）
    - `record_date`：记录日期（必填，字符串，格式YYYY-MM-DD，不能超过当前日期）

- **响应示例**：
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "id": "pat_123",
    "name": "张三",
    "gender": 1,
    "age": 65,
    "height": 175.00,
    "weight": 70.00,
    "phone": "13800138000",
    "created_at": "2024-05-20 14:30:00",
    "histories_count": 2,
    "histories": [
      {
        "id": "his_123",
        "type": 1,
        "content": "高血压病史10年",
        "record_date": "2024-01-01",
        "created_at": "2024-05-20 14:30:00"
      },
      {
        "id": "his_456",
        "type": 2,
        "content": "青霉素过敏",
        "record_date": "2024-01-01",
        "created_at": "2024-05-20 14:30:00"
      }
    ]
  },
  "request_id": "req_1234567890"
}
```

- **错误响应**：
  - **参数错误**：
  ```json
  {
    "code": 400,
    "msg": "参数错误",
    "data": {
      "errors": {
        "name": "患者姓名不能为空",
        "phone": "请输入正确的手机号码",
        "age": "年龄必须在1-120之间",
        "histories.0.record_date": "记录日期不能超过当前日期"
      }
    },
    "request_id": "req_1234567890"
  }
  ```
  
  - **权限不足**：
  ```json
  {
    "code": 403,
    "msg": "权限不足，无法创建患者信息",
    "request_id": "req_1234567890"
  }
  ```
  
  - **服务器错误**：
  ```json
  {
    "code": 500,
    "msg": "服务器内部错误",
    "request_id": "req_1234567890"
  }
  ```

- **设计说明**：
  - 接口支持批量创建患者基础病史，减少后续API调用次数
  - 病史内容支持HTML格式，便于富文本编辑器内容的存储和渲染
  - 所有必填字段都有明确的格式和范围限制
  - 错误响应包含详细的错误信息，便于前端展示和调试
  - 支持请求ID追踪，便于问题排查和日志分析
  - 使用事务确保患者信息和病史数据的一致性

### 2.2 时间轴事件接口

#### 2.2.1 获取时间轴事件列表
- **接口路径**：`/api/timeline-events`
- **请求方法**：`GET`
- **请求头**：
  - `Authorization`: `Bearer {token}`
- **请求参数**：
  - `patient_id`: 患者ID（必填）
  - `start_date`: 开始日期（可选，格式YYYY-MM-DD）
  - `end_date`: 结束日期（可选，格式YYYY-MM-DD）
  - `event_type`: 事件类型（可选，支持字符串类型：inspection(检查)、medication(用药)、treatment(治疗小结) 或数字类型：2(检查)、3(用药)、4(治疗小结)）
  - `page`: 页码（可选，默认1）
  - `page_size`: 每页数量（可选，默认20）
- **响应示例**：
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "list": [
      {
        "id": "evt_123",
        "patient_id": "pat_123",

        "event_type": "inspection", // 支持字符串类型：inspection(检查)、medication(用药)、treatment(治疗小结) 或数字类型：2(检查)、3(用药)、4(治疗小结)

        "content": "病情稳定，建议继续当前治疗方案",

}
```

## 2.4 事件类型对应关系

为确保前端与后端之间的一致性，特定义以下事件类型对应关系：

| 数字值 | 字符串值    | 事件类型   | 前端颜色标记 |
|--------|-------------|------------|--------------|
| 2      | inspection  | 检查事件   | 绿色#52c41a  |
| 3      | medication  | 用药事件   | 橙色#faad14  |
| 4      | treatment   | 治疗小结   | 蓝色#1890ff  |

**注**：数字值1（就诊）目前仅作为内部标识使用，前端不直接展示该事件类型。

## 3. 数据库设计

### 3.1 数据库表结构

#### 3.1.1 患者信息表（patient_info）
```sql
CREATE TABLE `patient_info` (
  `id` varchar(36) NOT NULL COMMENT '患者ID，UUID格式',
  `name` varchar(50) NOT NULL COMMENT '患者姓名',
  `gender` int(1) NOT NULL DEFAULT '0' COMMENT '性别（1:男 2:女）',
  `age` int(3) NOT NULL DEFAULT '0' COMMENT '年龄',
  `height` decimal(5,2) DEFAULT NULL COMMENT '身高（cm），范围50-250',
  `weight` decimal(5,2) DEFAULT NULL COMMENT '体重（kg），范围10-200',
  `phone` varchar(20) NOT NULL COMMENT '联系电话，11位手机号码',
  `created_by` varchar(36) NOT NULL COMMENT '创建人ID',
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_phone` (`phone`),
  KEY `idx_name` (`name`),
  KEY `idx_created_by` (`created_by`),
  KEY `idx_gender` (`gender`),
  KEY `idx_age` (`age`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='患者信息表';
```

#### 3.1.2 基础病史表（patient_history）
```sql
CREATE TABLE `patient_history` (
  `id` varchar(36) NOT NULL COMMENT '病史ID，UUID格式',
  `patient_id` varchar(36) NOT NULL COMMENT '患者ID，关联patient_info.id',
  `type` int(1) NOT NULL COMMENT '病史类型（1:既往病史 2:过敏史 3:家族病史）',
  `content` text NOT NULL COMMENT '病史内容，支持HTML格式',
  `record_date` date NOT NULL COMMENT '记录日期',
  `created_by` varchar(36) NOT NULL COMMENT '创建人ID',
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_patient_id` (`patient_id`),
  KEY `idx_type` (`type`),
  KEY `idx_record_date` (`record_date`),
  KEY `idx_created_by` (`created_by`),
  CONSTRAINT `fk_patient_history_patient` FOREIGN KEY (`patient_id`) REFERENCES `patient_info` (`id`) ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='基础病史表';
```

#### 3.1.3 时间轴事件表（timeline_event）
```sql
CREATE TABLE `timeline_event` (
  `id` varchar(36) NOT NULL COMMENT '事件ID',
  `patient_id` varchar(36) NOT NULL COMMENT '患者ID',
  `event_date` date NOT NULL COMMENT '事件日期',
  `event_type` int(1) NOT NULL COMMENT '事件类型（1:就诊 2:检查 3:用药 4:治疗小结）',
  `title` varchar(100) NOT NULL COMMENT '事件标题',
  `content` text NOT NULL COMMENT '事件内容',
  `is_key_node` int(1) NOT NULL DEFAULT '0' COMMENT '是否关键节点（0:否 1:是）',
  `created_by` varchar(36) NOT NULL COMMENT '创建人ID',
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_patient_id` (`patient_id`),
  KEY `idx_event_date` (`event_date`),
  KEY `idx_event_type` (`event_type`),
  KEY `idx_is_key_node` (`is_key_node`),
  CONSTRAINT `fk_timeline_event_patient` FOREIGN KEY (`patient_id`) REFERENCES `patient_info` (`id`) ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='时间轴事件表';
```

#### 3.1.4 附件表（attachment）
```sql
CREATE TABLE `attachment` (
  `id` varchar(36) NOT NULL COMMENT '附件ID',
  `event_id` varchar(36) NOT NULL COMMENT '事件ID',
  `file_name` varchar(200) NOT NULL COMMENT '文件名',
  `file_path` varchar(500) NOT NULL COMMENT '文件路径',
  `file_type` varchar(50) NOT NULL COMMENT '文件类型',
  `file_size` bigint(20) NOT NULL COMMENT '文件大小（字节）',
  `upload_by` varchar(36) NOT NULL COMMENT '上传人ID',
  `upload_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '上传时间',
  PRIMARY KEY (`id`),
  KEY `idx_event_id` (`event_id`),
  CONSTRAINT `fk_attachment_event` FOREIGN KEY (`event_id`) REFERENCES `timeline_event` (`id`) ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='附件表';
```

#### 3.1.5 用药记录表（medication_record）
```sql
CREATE TABLE `medication_record` (
  `id` varchar(36) NOT NULL COMMENT '用药记录ID',
  `event_id` varchar(36) NOT NULL COMMENT '事件ID',
  `drug_name` varchar(100) NOT NULL COMMENT '药物名称',
  `dosage` varchar(50) NOT NULL COMMENT '剂量',
  `usage` varchar(100) NOT NULL COMMENT '用法',
  `start_date` date NOT NULL COMMENT '开始用药日期',
  `end_date` date DEFAULT NULL COMMENT '结束用药日期',
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`id`),
  KEY `idx_event_id` (`event_id`),
  KEY `idx_drug_name` (`drug_name`),
  CONSTRAINT `fk_medication_event` FOREIGN KEY (`event_id`) REFERENCES `timeline_event` (`id`) ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用药记录表';
```

### 3.2 索引设计

#### 3.2.1 患者信息表索引
- 主键索引：`id`
- 唯一索引：`phone`（用于电话唯一性验证和快速查询）

### 3.3 数据完整性保障

1. **主键约束**：所有表都有唯一的主键，确保数据的唯一性和可识别性
2. **外键约束**：建立表之间的关联关系，确保数据的一致性和完整性
   - `patient_history.patient_id` → `patient_info.id`
   - `timeline_event.patient_id` → `patient_info.id`
   - `attachment.event_id` → `timeline_event.id`
   - `medication_record.event_id` → `timeline_event.id`
3. **唯一约束**：`patient_info.phone` 确保患者电话的唯一性
4. **非空约束**：关键字段设置为非空，确保数据的完整性
5. **默认值**：为部分字段设置合理的默认值，提高数据录入效率
6. **事务管理**：在创建或更新关联数据时使用事务，确保数据的一致性

### 3.4 数据安全设计

1. **访问控制**：通过API接口的权限控制，确保只有授权用户才能访问和操作数据
2. **数据加密**：敏感数据（如患者姓名、电话等）在传输和存储时进行加密
3. **日志记录**：记录所有关键操作的日志，便于审计和问题排查
4. **数据备份**：定期备份数据库，确保数据的安全性和可恢复性
5. **SQL注入防护**：使用参数化查询，防止SQL注入攻击
6. **XSS防护**：对用户输入的富文本内容进行过滤和转义，防止XSS攻击

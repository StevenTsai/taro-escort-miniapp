# 服务柔性化展示系统设计文档

## 1. 文档概述

### 1.1 文档目的
本文档描述了医疗陪护小程序服务展示系统的柔性化设计方案，旨在替换原有写死的服务展示逻辑，实现服务信息由后台配置、小程序端动态渲染的能力，支撑咨询、临床招募等新服务的快速上线。

### 1.2 适用范围
本文档适用于医疗陪护小程序的开发团队，包括后端开发、前端开发和测试人员。

### 1.3 术语定义
- **服务**：指平台提供的各种服务，如陪诊服务、咨询服务等
- **服务分类**：对服务进行归类的逻辑分组
- **服务流程**：服务的执行步骤和说明
- **排序权重**：控制服务和分类展示顺序的数值，越大越靠前

## 2. 功能需求

### 2.1 后台配置模块

#### 2.1.1 服务基础信息配置
- 支持新增/编辑/删除服务
- 配置项包括：服务名称、服务图标（100*100px）、服务价格（固定金额或"咨询客服"）、服务状态（启用/禁用）、排序权重

#### 2.1.2 服务流程配置
- 支持按步骤配置服务流程
- 每步包含：步骤名称、步骤描述、步骤图标
- 最多支持配置5步

#### 2.1.3 服务分类配置
- 支持创建服务分类（如陪诊服务、咨询服务、培训服务）
- 新增服务时可关联分类

### 2.2 小程序动态渲染模块

#### 2.2.1 服务列表页
- 按后台配置的分类分组展示
- 同一分类下按排序权重排序
- 展示服务名称、图标、价格及核心流程第一步描述
- 支持点击分类切换显示对应服务

#### 2.2.2 服务详情页
- 动态加载后台配置的服务名称、价格、完整流程（步骤图标+描述）、服务说明（富文本）
- 点击"立即预约"跳转至下单页，下单页字段与服务类型适配

### 2.3 兼容性要求
- 原有"全天陪诊""半天陪诊"等服务需迁移至新配置体系
- 确保历史服务正常展示及下单

## 3. 数据模型设计

### 3.1 服务分类表（service_category）

| 字段名 | 数据类型 | 描述 |
|-------|---------|------|
| id | VARCHAR(36) | 分类ID |
| name | VARCHAR(50) | 分类名称 |
| description | TEXT | 分类描述 |
| status | INT | 状态（1:启用 0:禁用） |
| sort_weight | INT | 排序权重 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 3.2 服务表（service）

| 字段名 | 数据类型 | 描述 |
|-------|---------|------|
| id | VARCHAR(36) | 服务ID |
| category_id | VARCHAR(36) | 分类ID |
| name | VARCHAR(100) | 服务名称 |
| icon | VARCHAR(255) | 服务图标URL |
| price_type | INT | 价格类型（1:固定金额 2:咨询客服） |
| price | DECIMAL(10,2) | 服务价格 |
| description | TEXT | 服务简介 |
| status | INT | 状态（1:启用 0:禁用） |
| sort_weight | INT | 排序权重 |
| content | TEXT | 服务说明（富文本） |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 3.3 服务流程表（service_process）

| 字段名 | 数据类型 | 描述 |
|-------|---------|------|
| id | VARCHAR(36) | 流程ID |
| service_id | VARCHAR(36) | 服务ID |
| step | INT | 步骤顺序 |
| name | VARCHAR(50) | 步骤名称 |
| description | TEXT | 步骤描述 |
| icon | VARCHAR(255) | 步骤图标URL |
| created_at | DATETIME | 创建时间 |

## 4. API接口设计

### 4.1 分类列表接口

#### 4.1.1 接口信息
- 接口路径：`/api/service-categories`
- 请求方法：`GET`
- 认证方式：无需认证

#### 4.1.2 响应示例
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "list": [
      {
        "id": "cat_1",
        "name": "陪诊服务",
        "sort_weight": 100
      },
      {
        "id": "cat_2",
        "name": "咨询服务",
        "sort_weight": 90

  }
}
```

## 6. 迁移方案

### 6.1 迁移策略
1. **数据迁移先行**：将现有写死的服务信息批量导入到新的数据表中
2. **保持ID一致性**：原有服务ID保持不变，确保历史订单关联正常
3. **分步迁移**：先迁移基础信息，再迁移服务流程
4. **灰度切换**：先并行运行，验证无误后完全切换到新系统

### 6.2 迁移步骤
1. **创建服务分类**：根据现有服务类型创建对应的分类
2. **导入服务基础信息**：将原有服务的名称、价格等信息导入到服务表
3. **配置服务流程**：为每个服务配置对应的流程步骤
4. **验证数据完整性**：检查所有服务信息是否正确导入
5. **切换前端调用**：修改小程序代码，使用新的API接口获取服务数据
6. **监控运行状态**：观察一段时间，确保系统稳定运行

### 6.3 兼容性保障
- 原有服务ID保持不变，确保历史订单查询正常
- 旧版小程序客户端兼容处理
- 图片加载失败的兜底处理

## 7. 技术实现要点

### 7.1 后端实现要点
- 使用RESTful风格设计API接口
- 服务流程最多支持5步，需在数据库层面限制
- 排序权重支持数值越大越靠前的排序逻辑
- 富文本编辑器支持常用格式和图片上传

### 7.2 前端实现要点
- 使用`Taro.request`调用API获取数据
- 服务卡片组件支持动态渲染价格类型（固定金额/咨询客服）
- 流程步骤组件支持最多5步的动态渲染
- 富文本内容使用`Taro.createRichText`或第三方组件渲染
- 分类切换使用`useState`管理当前选中分类
- 图片懒加载优化性能

### 7.3 后台配置端实现要点
- 服务图标和流程图标上传支持100*100px尺寸校验
- 表单验证确保必填字段完整性
- 提供服务数据导入导出功能
- 支持批量操作服务状态

## 8. 注意事项

### 8.1 性能优化
- 服务列表页使用分页加载，减少一次性请求数据量
- 所有图片使用CDN加速，确保小程序加载速度
- API接口添加缓存机制，减少数据库查询压力

### 8.2 安全性
- API接口添加适当的权限控制
- 图片上传添加文件类型和大小限制
- 富文本内容进行XSS过滤

### 8.3 可维护性
- 代码结构清晰，组件化设计
- 完善的注释和文档
- 统一的命名规范

### 8.4 扩展性
- 服务模型设计考虑未来可能的扩展字段
- API接口设计预留扩展参数
- 前端组件设计支持灵活配置

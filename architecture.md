# 医疗陪护小程序架构文档

## 1. 架构概述

医疗陪护小程序是一个基于Taro框架开发的跨平台应用，旨在为用户提供便捷的医疗陪护服务预约平台。本架构文档详细描述了系统的整体设计、技术选型、模块划分和数据流程，帮助开发团队理解和维护项目。

## 2. 技术栈选型

### 2.1 前端框架
- **Taro v4.1.5**：一套遵循React语法规范的多端统一开发框架，支持微信小程序、H5、支付宝小程序等多个平台
- **React 18**：用于构建用户界面的JavaScript库
- **NutUI-React-Taro v3.0.17**：基于React和Taro的UI组件库，提供丰富的移动端UI组件

### 2.2 状态管理
- **Zustand v5.0.6**：轻量级的状态管理库，提供简洁的API和优秀的性能

### 2.3 构建工具
- **Webpack 5.78.0**：模块打包工具（持久缓存 + Prebundle 已启用）
- **Babel**：JavaScript编译器，用于转换ES6+代码
- **Sass**：CSS预处理器

### 2.4 代码规范
- **ESLint**：代码质量检查（react-hooks + no-console 规则已启用）
- **Prettier**：代码格式化工具
- **husky + lint-staged**：Git pre-commit hooks

### 2.5 测试与CI
- **Jest**：测试框架（16 个套件、168 个用例）
- **Testing Library**：组件测试工具
- **GitHub Actions**：CI/CD 自动化（lint → test → build）

### 2.6 类型系统
- **TypeScript**：渐进式引入（tsconfig + 类型声明已配置）

## 3. 系统分层

系统采用经典的前端分层架构，主要分为以下几层：

### 3.1 用户层
直接与用户交互的界面层，包括微信小程序界面和H5页面等。

### 3.2 应用框架层
基于Taro和React构建的应用基础框架，提供跨平台开发能力和组件化开发支持。

### 3.3 应用组件层
包含页面组件、通用组件和自定义TabBar，实现具体的业务功能和UI展示。

### 3.4 状态管理层
使用Zustand实现全局状态管理，包括用户认证状态、TabBar状态、课程数据等8个store。

### 3.5 工具函数层
提供网络请求（拦截器+缓存）、认证处理（TypeScript）、图片处理（文件系统缓存+压缩）、统一支付、权限检查、订阅消息等工具函数。

### 3.5.1 常量层
TypeScript 常量模块，定义角色、订单状态、服务ID、颜色主题等全局常量，提供类型安全。

### 3.6 配置层
包含应用配置、环境配置等。

### 3.7 外部服务层
与后端API服务进行交互，获取和提交数据。

## 4. 核心模块设计

### 4.1 用户认证模块

**功能说明**：负责用户登录、注册、认证状态管理等功能。

**核心文件**：`src/store/useAuthStore.js`、`src/utils/auth.ts`、`src/constants/index.ts`

**主要功能点**：
- Token管理（条件添加，空Token不发送头）
- 粒度化登出（仅清除鉴权key，不影响城市偏好等数据）
- `isLoggedIn` 派生状态（`useIsLoggedIn` hook，避免状态不一致）
- JSON.parse 全部加 try/catch 防崩溃
- `setLoginInfo` 输入验证

**状态结构**：
```javascript
{
  token: string, // 用户认证Token
  userInfo: object, // 用户信息
  // isLoggedIn 为派生状态，通过 useIsLoggedIn() hook 获取
}
```

## 5. 数据流程

### 5.1 用户认证流程
1. 用户打开应用
2. 应用初始化时调用`useAuthStore`中的`initAuth`方法
3. 检查本地是否存在Token
4. 如果存在Token，验证Token是否有效
5. 根据验证结果更新用户认证状态
6. 用户进行登录或注册操作时，更新Token和用户信息

### 5.2 服务请求流程
1. 页面组件或业务组件调用API服务
2. `request.js`拦截请求，添加Token等认证信息
3. 发送请求到后端API
4. 接收响应，进行统一的响应处理
5. 如果响应成功，返回处理后的数据
6. 如果响应失败，根据错误类型进行相应的错误处理

### 5.3 状态更新流程
1. 用户交互或数据变化触发状态更新
2. Zustand store更新状态
3. 订阅该状态的组件自动重新渲染
4. 页面展示更新后的数据或UI
